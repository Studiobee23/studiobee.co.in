# Scope of Work Page (Quotes/Proformas) — Design Spec
**Date:** 2026-07-17

---

## Context

Quote and proforma PDFs generated by `mystudiobee` currently jump straight from the line items/totals/notes section to a hardcoded Terms & Conditions page. There is no dedicated place to describe what the engagement actually covers. We're adding a "Scope of Work" page, editable per document, positioned between the existing content and the Terms & Conditions page.

This follows the same additive pattern used for recent quote-editor fields (`line_item_view`, `summary_label`, `markup`, etc.): a new JSONB column, editor UI, and a template.ts render branch.

---

## Scope

- Applies to **quotes and proformas only** — same document types the Terms & Conditions page already targets (`doc.type === 'quote' || doc.type === 'proforma'`). Invoices and receipts are unaffected.
- Content is a **repeatable list of custom sections**, each with a heading and a body — not a fixed set of headings, not free-form single text block. The user defines however many sections they need per quote (e.g. "Deliverables", "Timeline", "Exclusions" — or anything else), and can add/remove/reorder them.
- Section body is **plain text** with line breaks preserved. No rich text formatting (bold/italic/bullets) in v1.
- The page **only renders when at least one section exists** — quotes with an empty `scope_of_work` array render exactly as they do today (no blank page inserted).
- **Not building in v1:** a reusable/default section template library (every quote starts blank, same as `notes` today). Can revisit if filling this in per-quote becomes tedious.

---

## Data Model

New migration (additive, non-destructive — matches `0016`–`0019` convention):

```sql
alter table documents add column if not exists scope_of_work jsonb not null default '[]';
```

Shape: `Array<{ heading: string; body: string }>` — structurally the same idea as `line_items`, just a simpler object per entry.

---

## Editor UI (`quote-editor.tsx`)

A new "Scope of Work" block, positioned near the existing Notes field, following the same list-editing conventions already used for line items and grouped-view reordering:

- `const [scopeOfWork, setScopeOfWork] = useState<{heading: string; body: string}[]>(doc?.scope_of_work ?? [])`
- Each row renders: a heading `<input>`, a body `<Textarea>`, a remove button (`Trash2` icon, matches line-item removal), and up/down `Chevron` buttons to reorder (matches the recent group-reorder feature)
- A "+ Add Section" button (`Plus` icon) appends a blank `{ heading: '', body: '' }` row
- `scopeOfWork` is included in the save payload alongside `line_items`, `notes`, etc.

---

## PDF Rendering (`template.ts`)

- `PdfDocument` type gains: `scope_of_work: Array<{ heading: string; body: string }>`
- New `.scope-page` block in `renderDocument()`, inserted in the HTML string between the `.notes-box` block (~line 401) and the `.terms-page` block (~line 405), giving this page order for quotes/proformas:

  `… → Notes → Scope of Work → Terms & Conditions`

- Renders only when:
  ```
  (doc.type === 'quote' || doc.type === 'proforma') && doc.scope_of_work?.length > 0
  ```
- Visual style matches `.terms-page`: `page-break-before: always`, same brand-blue title + horizontal rule header treatment (title text: "Scope of Work").
- Sections render **stacked in a single column** (not the 3-col grid `.terms-grid` uses) since section count and text length vary per quote — each section is a bold brand-blue heading followed by a body paragraph with `white-space: pre-line` to preserve the user's line breaks.
- All user-supplied text goes through the existing `esc()` helper before interpolation, consistent with every other user-editable field in the template (security rule from project CLAUDE.md).

---

## Out of Scope / Explicit Cuts

- No default/reusable section templates — blank per quote.
- No rich text formatting inside section bodies.
- No change to Terms & Conditions content or behavior.
- No changes to invoice/receipt rendering.
